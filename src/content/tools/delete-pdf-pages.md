## 1. The Mathematical Foundation: The PDF Tree Structure

To understand how a page is deleted, we must first look at the **Internal Hierarchy of the PDF Specification**.

### 1.1 The `/Pages` Dictionary and the `/Kids` Array
A PDF is not a linear list of pages; it is a "Tree."
- **The Node**: The root of the file is the `/Pages` object. It contains a pointer called `/Kids`, which is an array of IDs for the individual page objects.
- **The Deletion Step**: To delete page 5, the software must remove its ID from the `/Kids` array and recalculate the `/Count` attribute of the parent node.

### 1.2 Binary Stream Pruning
Once the page is unlinked from the tree, its "Resources" (the text streams, images, and fonts) are technically "Orphaned." 
- **The Optimization**: Professional-grade tools perform **Garbage Collection**. They scan the entire file to find any binary data associated *only* with the deleted page and physically remove those bytes from the disk, creating a "Lean and Clean" final document.

## 2. A Deep Dive into Xref Table Re-Indexing

Why is it so hard to just "Cut" a PDF? It's because of the **Cross-Reference Table**.

### 2.1 The Map of Byte Offsets
The Xref table lives at the end of the PDF and tells the viewer: "Object 10 starts at byte 500, Object 11 starts at byte 1000."
- **The Conflict**: When you delete a page and its binary data, every object *after* that point moves.
- **The Strategy**: Our engine performs a "High-Fidelity Re-Indexing," recalculating every byte offset from scratch and writing a clean, 100% compliant Xref table. This ensures your file opens instantly in every viewer in the world without "Corrupted File" errors.

### 2.2 Forensic Sanitization
